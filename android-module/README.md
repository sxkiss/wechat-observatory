# WeChat Observatory LSPosed Module

This is the companion module for `wechat-observatory`.

It runs inside WeChat through LSPosed and posts message DB insert events to the
gateway. The gateway stores WeChat events and queues admin-created send actions
for the module to receive and send from inside the WeChat process.

The current public compatibility target is WeChat for Android `8.0.75`.
WeChat internals change between releases; validate message observation, contact
sync, media upload, and Action Outbox sends after upgrading WeChat.

## Hook Point

The module scopes to `com.tencent.mm` and hooks:

```text
com.tencent.wcdb.database.SQLiteDatabase.insertWithOnConflict(
  String table,
  String nullColumnHack,
  ContentValues values,
  int conflictAlgorithm
)
```

For `table == "message"`, it reads:

- `talker`
- `content`
- `isSend`
- `msgId`
- `createTime`
- `type`

Then it posts an event to:

```text
POST <bridge_url>/webhook/lsposed/message
```

Text messages are posted as their normalized text. Non-text message rows are
also posted with a stable placeholder such as `[图片]`, `[语音]`, `[视频]`,
`[表情]`, `[位置]`, or `[链接/文件]`, and WeChat's numeric `message_type` is
preserved. When the module can find the underlying WeChat media file, it uploads
`media_base64` plus `media_kind`, `media_mime`, `media_name`, and `media_size`
so the gateway can render an actual preview or player.

## Runtime Config

The module now includes a launcher settings page. After installing and enabling
the LSPosed module, open **WeChat Observatory** on the phone, fill in the server
URL and API Key generated by the Web admin console, then save and restart
WeChat.

The settings page writes the same keys exposed by `BridgeConfigProvider`, so
the hooked WeChat process reads the values without manually editing files.

Create this file on the phone:

```text
/data/data/com.tencent.mm/files/wechat-observatory/config.properties
```

The module keeps `/sdcard/Download/wechat-observatory/config.properties` and the
old `/sdcard/Android/media/cc.wechat.observatory/config.properties`
path as fallbacks, but Android scoped-storage rules can prevent WeChat's
process from reading external storage config files on recent Android versions.

Example:

```properties
enabled=1
bridge_url=http://192.168.1.10:8088
api_key=wg_dev_key
poll_interval_ms=1000
poll_limit=4
outbox_parallelism=2
contact_sync_interval_ms=600000
contact_sync_limit=1000
contact_include_chatrooms=1
media_upload_enabled=1
media_upload_limit_bytes=5242880
```

Restart WeChat after changing config.

`poll_interval_ms` controls how often the module checks the gateway outbox.
The worker currently sleeps at least 1000ms between polls, so values below
1000 do not make sends faster.

`poll_limit` controls how many leased outbox items the module requests per wake.
The gateway currently caps one batch at 4 items.

`outbox_parallelism` controls how many outbox lanes the module may execute in
parallel. The dispatcher keeps the same `wxid + kind` lane serialized so
message DB verification does not race on the same target and message type.

On Android builds where the WeChat process cannot resolve the module config
provider, the module log can show `read config from provider returned null
cursor`. In that case, update the fallback `config.properties` file that WeChat
can read, such as
`/sdcard/Android/data/com.tencent.mm/files/wechat-observatory/config.properties`.
Changing only the module app's SharedPreferences will not affect the hooked
WeChat worker in that state.

`contact_sync_interval_ms=0` disables friend sync. The default is 10 minutes.
`contact_sync_limit` caps one snapshot upload. `contact_include_chatrooms=1`
includes chatrooms and is the default.

## Outbox Protocol

When WeChat loads, the module:

1. Registers the current WeChat identity with `/module/register` when
   `api_key` is configured and the current WeChat `wxid` has been detected.
2. Opens `GET /module/outbox/ws?api_key=...&device=...&wxid=...`.
3. Receives `outbox` messages as soon as the gateway queues reply actions.
4. Attempts to send each returned item from inside WeChat.
5. Reports results with a WebSocket `ack` message.

If the WebSocket connection cannot be established or drops, the module falls
back to `POST /module/outbox/poll` and `POST /module/outbox/ack`. The fallback
uses the same MySQL outbox state machine, so queued replies are not lost.

The outbox WebSocket loop runs on its own daemon worker. The main worker keeps
registration, contact sync, and inbound message polling fallback alive while
the outbox socket is connected.

The send adapter is intentionally isolated behind Action Outbox dispatch in
`HookEntry`. Text still uses the lower-level builder path on WeChat Android
`8.0.75`:

```text
w11.s1.a(toUser) -> w11.r1.g/e/h(...) -> w11.r1.a().a()
```

If that builder path is unavailable, it falls back to
`tg3.t1.a() -> dk5.s5.fj(...)`. Live validation showed that path can hit
WeChat's service-locator initialization guard from the module worker thread, so
the builder is the safer primary path. Both paths are WeChat-process calls; the
module does not parse or execute commands.

Action Outbox v1 dispatches `text`, `image`, `video`, `voice`, `file`, `emoji`,
`location`, `quote`, `link`, `mini_program`, and `chat_history` from the gateway
queue. Unsupported or malformed kinds must ACK `failed` instead of being
silently skipped. Payment-like and system-only message types are observation
boundaries, not send actions.

On WeChat Android `8.0.75`, the installed app can run through Tinker. The send
adapter must resolve WeChat classes with the patched runtime loader returned by
`Application.getClassLoader()`, which can be a `DelegateLastClassLoader` over
`/data/user/0/com.tencent.mm/tinker/.../tinker_classN.apk`. Using the base
`PathClassLoader` can load duplicate native libraries and leave WeChat kernel
services uninitialized.

After changing WeChat versions, validate the send paths on a test account
before using them with real conversations. The module should ACK `sent` only
after the WeChat-process send call returns without an error.

The send path currently returns a successful ACK without `chat_record_id`
because the WeChat send calls do not return it. Precise local message-id
correlation can be added later by matching the subsequent self-sent message DB
insert.

## Contact Snapshot

The module remembers the WeChat WCDB object from `message` or `rcontact` table
writes. On the worker loop it best-effort reads:

```sql
SELECT username,nickname,conRemark,alias,type,verifyFlag
FROM rcontact
WHERE username IS NOT NULL AND username <> ''
LIMIT ?
```

It uploads ordinary friends plus the special `filehelper` conversation to:

```text
POST <bridge_url>/module/contacts/snapshot
```

with payload:

```json
{
  "api_key": "wg_dev_key",
  "device": "phone-a",
  "wxid": "wxid_current",
  "complete": true,
  "contacts": [
    {
      "wxid": "wxid_friend",
      "nickname": "Friend",
      "remark": "Remark",
      "alias": "alias",
      "type": 3,
      "verify_flag": 0,
      "chatroom": false,
      "deleted": false
    }
  ]
}
```

If the database has not been captured yet, or the query returns no friends, the
module skips upload and keeps message/outbox behavior running. `filehelper` is
kept even when WeChat stores it as a system/non-friend row, and a blank nickname
is reported as `文件传输助手`. The module does not parse or execute business
commands.

When reinstalling the module under the current Vector/LSPosed runtime, verify
the module row in `/data/adb/lspd/config/modules_config.db` still points at the
new installed `base.apk`, keep the scope on `com.tencent.mm`, and reboot before
validating send behavior. Android changes the APK path after reinstall.
